# Project Decisions

> This file captures project-specific decisions that agents should reference
> when working on this design system. After making a major design or architectural
> decision, document it here so all agents can apply it consistently going forward.
>
> **For agents:** Read this file before starting work. Apply every documented decision
> to your output. After completing a task that involved a repeatable choice, suggest
> the user add it here.

## How to Use This File

When you make a significant, repeatable decision about your design system — such as
choosing a color palette, defining component API patterns, or establishing spacing
conventions — add it below. Agents will read this file and apply your decisions
consistently across all future work.

Use this format for each decision:

```
### [Category]: [Decision Title]
**Decision:** What you decided
**Context:** Why — what problem it solves or what constraint it addresses
**Applies to:** Where this decision should be enforced (tokens, components, SCSS, etc.)
```

### What belongs here

- Color palette, typography, or spacing scale choices
- Component API patterns (prop naming conventions, variant sets, size scales)
- Accessibility patterns beyond WCAG minimums
- Animation and motion conventions
- Icon system choices (library, default sizes, usage rules)
- Responsive breakpoint definitions
- Error, validation, and empty state display patterns
- Content and copy conventions (tone, terminology, truncation rules)
- Any "we always do it this way" convention specific to your project

### What does NOT belong here

- Starter kit architectural decisions (those are in `docs/ARCHITECTURE.md` Section 13)
- One-off bug fixes or temporary workarounds
- Implementation details that only apply to a single component
